1 Introduction

Les DOI (“Digital Object Identifier System” 2018) permettent d'identifier des articles, documents, livres, en ligne ou non et de façon unique (un peut comme les ISBN pour les livres). Ils sont uniques et délivrés par un organisme: http://www.doi.org

On retrouve ces DOI sur des références à de nombreux documents (notamment scientifiques). Par exemple, pour une recherche de "tim berners lee" sur le moteur http://search.crossref.org/ on obtiendra le résultat suivant (parmi d'autres):

  1. En suivant l'URL https://doi.org/10.1145/147126.147133 (correspondant au DOI) dans le browser, nous sommes redirigés vers la page de l'éditeur de l'article.
  2. En demandant une autre forme de contenu, en RDF (“RDF 1.1 Primer” 2018), sur la même URL on obtient des informations directement exploitable par un programme.

On observe l'entête Link: <http://dx.doi.org/10.1145/147126.147133>; rel="canonical",...: elle donne l'url préférée (canonique) du document que je suis en train de récupérer. C'est cette URL qui va me servir d'URI pour identifier le sujet (souvenez vous: "sujet" - "prédicat" - "objet") de la resource qui m'intéresse dans le document: ici <http://dx.doi.org/10.1145/147126.147133>

Et c'est ce sujet qui va servir à trouver le titre: ce titre correspond à l'objet de la relation:

<http://dx.doi.org/10.1145/147126.147133>
         <http://purl.org/dc/terms/title> "WWW" .

soit: WWW.

Le vocabulaire utilisé pour le prédicat est: http://purl.org/dc/terms/ (“DCMI: DCMI Metadata Terms” 2018), il est décrit à l'adresse: http://www.dublincore.org/documents/dcmi-terms/

On s'aperçevra que les DOI sont tous disponibles (à ce jour, a priori mais ça peut changer) sous la forme: http://dx.doi.org/{IDENTIFIANT}

2 Réalisation

L'objet du projet est d'afficher des informations pour n'importe quel DOI.

  1. Fournir un micro-service web qui dispose d'un formulaire pour lequel on saisit l'IDENTIFIANT et, à sa soumission, retourne un contenu HTML qui contient <a href="http://dx.doi.org/{IDENTIFIANT}>{Le titre du document}</a> - {La date du document}
    • La date peut être trouvée en utilisant dcterms:date ou dcterms: correspond au prefix http://purl.org/dc/terms/
  2. Dans un second temps, en plus du lien du titre et de la date, afficher les auteurs sous la forme d'une liste de noms (vocabulaire FOAF (“FOAF Vocabulary Specification” 2018)).
  3. PRODUCTION BONUS (des points en plus, pas en moins :-): produire le même développement pour afficher la liste des noms tels que disponibles sur http://viaf.org/: Si vous vous y êtes bien pris, très peu de lignes de code sont différentes.

3 Guide de résolution

3.1 Choix des technologies

  • Vous avez le choix de réaliser le dispositif en ruby, python, php ou javascript.

Vous trouverez pour chacun des languages, des bibliothèques qui vous permettent de faire des requêtes et de manipuler des documents RDF (il en existe d'autres, voici celles que je vous conseille):

langage bibliothèque facilité d'utilisation installation
Ruby linkeddata Très Facile gem install linkeddata
PHP EasyRDF Easy http://www.easyrdf.org/docs/getting-started
Python rdflib Très facile pip install rdflib
Javascript rdflib.js Moins facile (mal documenté) npm install rdflib

Pour les micro-framework je préconise:

  • pour ruby: sinatra
  • pour python: flask
  • pour php: slim
  • pour javascript: expressjs

3.2 Tâches à executer

  1. Fournir un formulaire pour saisir l'IDENTIFIANT
  2. à la soumission du formulaire:
    1. construire l'url du document http://dx.doi.org/{IDENTIFIANT}
    2. construire un graph RDF avec la bibliothèque en chargeant le contenu du document à l'adresse construite
    3. parcourir le graphe pour récupérer le titre et la date (il faut fixer le sujet et le prédicat pour sélectionner les objets qui correspondent)
    4. afficher le résultat.
  3. Ajouter les autheur

3.3 Conseils

  • Essayez d'abord de manipuler le fichier turtle fourni avant d'utiliser des resources en lignes: ça vous permettra de distinguer les problèmes de conceptions des problèmes de connexion....
  • D'autres DOI sont disponibles sur crossref.

En théorie, ces opérations peuvent être effectuées côté client, mais en pratique, peut de serveurs authorisent les "CORS". De plus les bibliothèques javascript ne sont pas facile d'accès. C'est pourquoi je vous demande d'effectuer ces traitements côté serveur.

4 Livrables

Un fichier zip contenant:

  • le code de l'application
  • un fichier docker-compose.yml, tel qu'il n'y ai plus qu'a faire docker-compose up pour lancer le serveur, disponible sur le port 3000 de la machine locale.
  • un readme qui explique comment fonctionne votre production (type github), avec votre nom, prénom et adresse e-mail.
